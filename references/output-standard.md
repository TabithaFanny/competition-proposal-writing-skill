# 成稿输出规范 / Output Standard

> 定义章节写作完成后的文件输出格式、图表存放位置、命名规则。
> 确保多个章节并行或交叉写作时，输出物整齐可归、可追溯。

---

## 零、章节处理的标准流程（review → 改写 → 成稿）

### 流程说明

每章节默认按以下顺序自动执行，不需用户确认：

```
第一步：结构化 review
  → 本章整体功能定位
  → 结构完整性评估
  → 逐小节诊断（具体到哪句、什么问题、严重程度、如何补强）
  → 问题优先级汇总（P0/P1/P2）
  → 改写策略说明
  → 数据核验清单

第二步：research 补证据（针对 review 发现的数据缺口）
  → 数据存疑时，先全网搜索核验
  → 找到可靠来源后直接补入并标注
  → 找不到时：改保守表述 / 标注"待核实" / 删除
  → 不再停下来等用户确认

第三步：正式改写输出
  → 按 review 结果和改写策略执行
  → 以"信息密度、逻辑性、真实性、正式性"为优先
  → 扩写必须增加有效信息，不能注水

第四步：输出后自检
  → 是否只写长、还是增强了论证
  → 是否新增有效信息维度
  → 是否存在未核实却写得很确定的数据
  → 是否保持整章逻辑一致性
```

### review 的粒度要求

每条 review 必须尽量具体，指向"如何改"：

```
❌ 错误示例（泛泛点评）：
"2.1.3 社会环境可以加强数据支撑"

✅ 正确示例（具体可执行）：
"第31行'21.1%老龄占比'，数据偏旧，应更新至2024年数据（约21.8%），
  来源：国家统计局2024年国民经济公报。如无法查到，改为'持续上升'
  或标注'待核实'，不能直接沿用2023年数据。"
```

### 两种输出模式的区分

| 模式 | 适用场景 | 要求 |
|------|----------|------|
| **分析稿 / 审阅稿** | review 报告、数据核验清单、图表清单 | 可以结构化、分点、列清单，方便审查 |
| **成稿 / 交付稿** | 正式比赛项目书正文 | 必须段落化、连续成文、自然过渡，少用 bullet points |

### 成稿的文风要求

最终交付的正文必须像"人写的项目书"，不像"AI 整理的提纲"：

```
❌ 过度分点化的正文（像提纲）：
"三大核心优势：
一、效率提升...
二、成本降低...
三、风险管控..."

✅ 段落化的正式正文：
"本项目的核心优势体现在效率、成本和风险管控三个维度。
在效率层面..."

❌ 机械分点（像审阅记录）：
"问题：数据缺来源
建议：补充来源
优先级：P0"

✅ 自然段落（像项目书）：
"市场规模数据因来源机构不同存在口径差异，需在修订版中统一标注..."
```

### 成稿模式下的默认规则

- 能用自然段讲清楚的，不要拆成分点
- 能融入正文的内容，不要强行做表格
- 每段表达一个完整意思
- 段与段之间要有承接
- 结构存在于内部逻辑，不要过度暴露成"分点感"

### 适合保留结构化的内容

以下内容允许保留结构化形式：
- review 报告
- 数据核验清单
- 图表清单
- 执行说明
- 附录材料

## 一、两阶段输出流程

当用户说「第X章全部输出」时，默认执行两阶段流程：

### 阶段一：正文先行
1. 读取 `.project-config.md` 获取项目路径
2. 输出成稿正文，正文中使用**图位占位符**：
   ```
   图4-1【待生成】：市场规模三层筛选模型
   图4-2【待生成】：核心竞品能力对比图
   ```
3. 同时输出可视化清单，标注优先级：
   - **必做**：没有它正文明显变弱
   - **建议做**：有了更好，但没它也能交
   - **可选**：主要用于答辩增强
4. 等待用户确认清单

### 阶段二：批量生成
5. 用户确认后，一次性生成全部图表
6. 输出最终汇总包

### 核心原则
- 正文逻辑先于图表生成
- 图表必须服务正文，不允许脱离正文单独堆图
- 如果某张图不如文字清晰，保留文字，不强制生成

## 二、图位占位符规范

正文中的图位占位符格式：

```
图{章节}-{序号}【待生成】：{图表名称}
```

示例：
```
图4-1【待生成】：系统分层架构图
图4-2【待生成】：数据流转闭环图
图4-3【待生成】：三级风险分级对照表
```

生成图表后，将占位符替换为实际引用：
```
图4-1：系统分层架构图（来源：assets/charts/market/tech-architecture-v1.html）
```

## 三、章节文件命名

### 命名格式

```
chapter-{序号}-{章节简称}.md
```

### 示例

| 章节 | 文件名 |
|------|--------|
| 第一章 执行摘要 | `chapter-01-executive-summary.md` |
| 第二章 市场与需求分析 | `chapter-02-market-analysis.md` |
| 第三章 产品机制分析 | `chapter-03-product-analysis.md` |
| 第四章 技术分析 | `chapter-04-technical-analysis.md` |
| 第五章 应用空间与推广策略 | `chapter-05-market-expansion.md` |
| 第六章 运行保障 | `chapter-06-operations.md` |

### 存放位置

```
项目根目录/

```

## 四、图表目录结构

### 目录命名

```
assets/charts/{章节分类}/
```

| 章节分类 | 对应章节 | 示例 |
|----------|----------|------|
| market | 市场分析、竞争格局 | `assets/charts/market/` |
| product | 产品介绍、功能架构 | `assets/charts/product/` |
| financial | 财务数据、收入模型 | `assets/charts/financial/` |
| competitors | 竞品对比 | `assets/charts/competitors/` |
| team | 团队介绍 | `assets/charts/team/` |
| operation | 运营数据 | `assets/charts/operation/` |
| risk | 风险分析 | `assets/charts/risk/` |

### 图表文件命名

```
{图表类型}-{版本号}.{后缀}
```

| 类型 | 后缀 | 说明 |
|------|------|------|
| HTML结构图 | `.html` | 架构图、流程图、关系图 |
| Python图表 | `.png` | 数据图、柱状图、饼图、表格 |
| Python生成脚本 | `.py` | matplotlib源代码 |
| 低保真文档 | `-LowFi.md` | 生成前的设计方案 |

### 示例

```
assets/charts/market/
├── tech-architecture-v1.html        # 系统分层架构图
├── data-flow-v1.html               # 数据流转闭环图
├── hybrid-decision-v1.html         # 混合方案决策流程图
├── risk-levels-table.py            # 三级风险分级表（脚本）
├── risk-levels-table.png           # 三级风险分级表（输出图）
├── tam-sam-som-funnel-v1.png      # 市场规模漏斗图
└── positioning-triangle-v3-LowFi.md # 定位三角低保真文档
```

## 五、可视化建议格式

章节末尾的「可视化建议」统一使用以下格式：

```markdown
## 可视化建议

| 位置 | 建议作图 | 类型 | 优先级 | 核心信息 |
|------|----------|------|--------|----------|
| 4.1.1 | 系统分层架构图 | HTML | 必做 | 四层架构（终端→业务→能力→数据）及关键技术栈 |
| 4.1.2 | 数据流转图 | HTML | 必做 | 居民→AI分析→工单生成→社工处理→统计反哺完整闭环 |
| 4.2.3 | 混合方案决策流程图 | HTML | 必做 | 规则引擎与大模型的协同决策路径及降级逻辑 |
| 4.3.2 | 三级风险分级对照表 | Python | 必做 | 高/中/低风险判定条件、处置方式、SLA要求对比 |
| 4.4 | 系统容错与安全架构图 | HTML | 建议做 | 容错设计三层保障 + 隐私合规全链路 |
```

### 字段说明

| 字段 | 说明 |
|------|------|
| 位置 | 对应章节的4.1.1、4.2.3等小节编号 |
| 建议作图 | 图表的中文名称，需足够具体 |
| 类型 | `HTML` 或 `Python`，决定生成工具 |
| 优先级 | `必做` / `建议做` / `可选` |
| 核心信息 | 一句话说明图表要传达的核心结论 |

## 六、生成优先级

当多个图表需要生成时，按以下优先级处理：

```
1. 结构型内容 → HTML优先
   - 架构图、流程图、关系图
   - 需要展示元素间关系的

2. 数据型内容 → Python优先
   - 市场规模、占比、对比
   - 需要精确数值展示的

3. 概念型内容 → MiniMax图片生成
   - 场景图、产品mockup
   - 暂未开通，标记待补充
```

## 七、输出检查清单

章节完成后，检查以下事项：

```
□ 章节文件已保存至项目根目录
□ 文件名符合 chapter-{序号}-{章节简称}.md 格式
□ 可视化建议已附在章节末尾
□ 每个可视化建议已生成对应文件
□ HTML图表已保存至 assets/charts/{分类}/
□ Python图表已保存至 assets/charts/{分类}/
□ Python脚本已保存至 assets/charts/{分类}/*.py
□ 图表引用路径已在正文中标注（如 图4-1）
```
