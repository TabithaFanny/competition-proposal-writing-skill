# 成稿输出规范 / Output Standard

> 定义章节写作完成后的文件输出格式、图表存放位置、命名规则。
> 确保多个章节并行或交叉写作时，输出物整齐可归、可追溯。

---

## 零、章节处理的标准流程（review → 改写 → 成稿）

### 触发条件

当用户要求：
- review
- 帮我检查
- 先review再改
- 逐章修订
- 系统处理全部章节
- 不要直接改，先做诊断

默认启用"章节 review → 逐章修订"流程。详细执行规范见：[review-rewrite-protocol.md](references/review-rewrite-protocol.md)

### 流程说明

每章节默认按以下顺序自动执行，不需用户确认：

```
第一步：结构化 review
  → 本章整体功能定位
  → 结构完整性评估
  → 逐小节诊断（具体到哪句、什么问题、严重程度、如何补强）
  → 问题优先级汇总（P0/P1/P2）
  → 改写策略说明
  → 数据核验清单

第二步：research 补证据（针对 review 发现的数据缺口）
  → 数据存疑时，先全网搜索核验
  → 找到可靠来源后直接补入并标注
  → 找不到时：改保守表述 / 标注"待核实" / 删除
  → 不再停下来等用户确认

第三步：正式改写输出
  → 按 review 结果和改写策略执行
  → 以"信息密度、逻辑性、真实性、正式性"为优先
  → 扩写必须增加有效信息，不能注水

第四步：输出后自检
  → 是否只写长、还是增强了论证
  → 是否新增有效信息维度
  → 是否存在未核实却写得很确定的数据
  → 是否保持整章逻辑一致性
```

### review 的粒度要求

每条 review 必须尽量具体，指向"如何改"：

```
❌ 错误示例（泛泛点评）：
"2.1.3 社会环境可以加强数据支撑"

✅ 正确示例（具体可执行）：
"第31行'21.1%老龄占比'，数据偏旧，应更新至2024年数据（约21.8%），
  来源：国家统计局2024年国民经济公报。如无法查到，改为'持续上升'
  或标注'待核实'，不能直接沿用2023年数据。"
```

### 两种输出模式的区分

| 模式 | 适用场景 | 要求 |
|------|----------|------|
| **分析稿 / 审阅稿** | review 报告、数据核验清单、图表清单 | 可以结构化、分点、列清单，方便审查 |
| **成稿 / 交付稿** | 正式比赛项目书正文 | 必须段落化、连续成文、自然过渡，少用 bullet points |

### 成稿的文风要求

最终交付的正文必须像"人写的项目书"，不像"AI 整理的提纲"：

```
❌ 过度分点化的正文（像提纲）：
"三大核心优势：
一、效率提升...
二、成本降低...
三、风险管控..."

✅ 段落化的正式正文：
"本项目的核心优势体现在效率、成本和风险管控三个维度。
在效率层面..."

❌ 机械分点（像审阅记录）：
"问题：数据缺来源
建议：补充来源
优先级：P0"

✅ 自然段落（像项目书）：
"市场规模数据因来源机构不同存在口径差异，需在修订版中统一标注..."
```

### 成稿模式下的默认规则

- 能用自然段讲清楚的，不要拆成分点
- 能融入正文的内容，不要强行做表格
- 每段表达一个完整意思
- 段与段之间要有承接
- 结构存在于内部逻辑，不要过度暴露成"分点感"

## 零-1、文体约束（正式修订稿规范）

### 核心定位

正式修订稿应接近"论文正文/项目申报书正文"的连贯写法。优先使用连续书面段落，不使用展示型框架语言。

正式修订稿默认采用项目申报书/正式计划书正文体，而不是演示稿、路演稿、答辩稿、宣传稿或提纲扩写稿。

### 八条基本规则（必须遵守）

1. **review 可以结构化呈现**，但修订稿必须是可直接进入正式文稿的连续正文
2. **除章标题、节标题外**，默认不再继续添加提示性三级标题或标签式小标题
3. **禁止面向写作者或评委的元说明**："章节定位"、"评委只会看几分钟"、"本节决定第一印象"等
4. **禁止演示型、提纲型小标题**："场景切入"、"核心价值"、"三维痛点分析"、"竞品差异化优势"等，除非用户明确要求保留
5. **禁止提纲扩写句式**："第一是……第二是……第三是……"应改写为自然连贯的正式段落
6. **禁止短句堆叠**：展示感过强、口号化明显的文本不允许
7. **段落论证结构**：每一节正文应以完整段落展开，优先形成"论点—解释—证据—结论"的书面论证结构
8. **可直接交付**：输出结果应让用户可以直接复制到项目书正文中，无需再进行二次去提纲化处理

### 禁止的文风

| 禁止类型 | 示例 | 说明 |
|---------|------|------|
| 演示稿/路演稿腔 | "评委看到的第X部分，决定第一印象"、"2分钟浏览本章" | meta叙事，评委不会想看到这类写作提示 |
| 提纲式标题 | "场景切入"、"核心价值"、"三维痛点"、"项目定位"、"第一是…第二是…" | 过度的结构暴露，破坏正文连贯性 |
| 口号化语言 | "被动响应→主动预防"、"经验驱动→数据驱动"、"早发现、早介入、早化解" | 过度使用政策语言和政治表态 |
| 宣传稿化 | "令人振奋"、"重磅"、"刚刚" | 情绪化表达，不适合正式申报书 |
| 分点堆砌 | 连续使用"第一…第二…第三…"结构 | 破坏段落连贯性 |
| 短句堆叠 | 连续独立短句，缺乏论证展开 | 展示感过强 |

### 允许保留的标题类型

- 章节级标题：`1.1 项目简介`、`2.3 市场容量`（阿拉伯数字编号）
- 节内小标题：仅在确实需要切换主题时使用，且应简洁（不超过8字）
- 禁止添加提示性、标签性小标题如"场景切入"、"痛点分析"、"核心优势"等

### 段落写作标准

1. **论点-解释-证据-结论**结构：每段首先亮出论点，然后解释，最后给出证据或举例
2. **自然过渡**：段与段之间通过逻辑连接词衔接，不用"下面"、"首先"、"其次"等机械过渡
3. **长度控制**：每段不少于100字，不超过400字；信息密集时可更长
4. **数据融入**：数据不是独立展示，而是作为论据融入段落论证中

### review 与修订稿的文体切换

- **review 输出**：可以结构化、分点、列清单（供审查用的诊断语言）
- **修订稿输出**：必须转换为连贯的正式正文语言，不能延续 review 的框架
- **切换检查**：修订稿中不得出现"问题如下"、"建议"、"P0/P1/P2"、"待核实"等诊断性词汇

### 正确示例（参考风格）

```
2.1.1 政策环境：基层治理现代化已上升为国家战略，智慧社区与人工智能应用进入加速落地期

近年来，基层治理现代化已不再只是基层工作的单项要求，而是被纳入国家治理体系和治理能力现代化的整体部署之中，成为推进中国式现代化的重要基础工程。2021年，中共中央、国务院印发《关于加强基层治理体系和治理能力现代化建设的意见》，明确提出"到2035年，基层治理体系和治理能力现代化基本实现"的总体目标，并强调要提高基层治理的社会化、法治化、智能化、专业化水平。这标志着基层治理现代化首次以中长期国家目标的形式被系统提出，其政策含义已超越传统意义上的社区管理和社会服务优化，而上升为国家治理现代化的重要支撑领域。

从治理基础看，我国基层治理体系已具备较强的组织覆盖和制度基础，但同时也面临服务精细化不足、数据协同不畅、专业队伍支撑薄弱等现实瓶颈。根据《"十四五"城乡社区服务体系建设规划》披露的数据，截至2020年底，全国已有50.9万个村民委员会、11.3万个居民委员会初步实现信息集中汇聚、统一管理、动态更新；城市社区综合服务设施已实现全覆盖，农村社区综合服务设施覆盖率达到65.7%；全国社区工作者规模达到433.8万人，村（居）民委员会班子成员达277.6万人，社会工作专业人才达157.3万人。这说明我国基层治理已形成较大规模的组织体系与服务网络，但也意味着，随着治理对象、治理事项和群众需求快速增长，仅靠传统人工分办、经验处置和碎片化信息系统，已难以支撑高频、复杂、跨部门的基层治理任务。
```

### 自检清单

```
□ 全文无不必要的小标题（章节编号标题除外）
□ 每段均为连贯段落，无分点堆砌
□ 无meta叙事（如"评委看到"、"本章"等）
□ 无口号化语言（"早发现、早介入"等三字结构）
□ 数据融入正文论证，非独立列示
□ review词汇已清除（问题/建议/P0等）
□ 段落间有自然过渡
□ 无短句堆叠、无展示感过强
□ 可直接复制到项目书正文，无需二次处理
```

## 零-2、禁止 meta 表述（含 HTML 输出）

### 核心原则

无论输出为普通文本还是 HTML，均禁止输出任何 meta 表述、过程播报、执行说明、过渡提示或面向写作者/读者的解释性语言。

### 禁止内容清单

| 禁止类型 | 示例 | 说明 |
|---------|------|------|
| 说明自己要做什么 | "现在开始重写"、"下面是修订版本"、"接下来我将……" | 过程播报 |
| 解释内容功能 | "本节用于说明……"、"这一部分主要展示……" | 用途说明 |
| 面向读者的提示 | "评委会重点关注……"、"本章决定第一印象……" | 面向评委/读者的meta语言 |
| 写作过程说明 | "对写作过程、结构选择、修改策略的说明" | 解释性语言 |
| 引导性套话 | "以下是……"、"如下所示……"、"这里展示的是……" | 过渡提示 |

### HTML 输出的额外限制

- 不得在 HTML 注释中加入 meta 表述
- 不得在 `<p>`、`<div>`、`<section>`、`<aside>`、`<span>`、`<header>`、`<footer>` 等标签中加入过程说明、用途说明或提示性文案
- 不得使用 note、tip、intro、summary、caption 等说明性区块承载 meta 内容，除非用户明确要求
- 不得输出"过渡段"、"说明段"、"引导段"来介绍接下来的正文
- HTML 内容必须直接进入实际交付内容本身，而不是先解释再输出

### 判断标准

如果一段文字删掉后不影响正文信息本身，只是在解释"这是什么""接下来要干什么""为什么这样写"，则该内容属于 meta 表述，必须删除。

### 正确示例 vs 错误示例

```
❌ 错误（meta表述）：
"下面是第三章的修订版本，我们消除了分点结构。"

✅ 正确（直接输出）：
（直接输出修订后的正文，不做任何说明）

❌ 错误（HTML中的meta）：
<!-- 本节展示系统架构图，用于说明技术方案 -->

✅ 正确（HTML中直接内容）：
<section>系统架构图</section>
```

### 适合保留结构化的内容

以下内容允许保留结构化形式：
- review 报告
- 数据核验清单
- 图表清单
- 执行说明
- 附录材料

## 一、两阶段输出流程

当用户说「第X章全部输出」时，默认执行两阶段流程：

### 阶段一：正文先行
1. 读取 `.project-config.md` 获取项目路径
2. 输出成稿正文，正文中使用**图位占位符**：
   ```
   图4-1【待生成】：市场规模三层筛选模型
   图4-2【待生成】：核心竞品能力对比图
   ```
3. 同时输出可视化清单，标注优先级：
   - **必做**：没有它正文明显变弱
   - **建议做**：有了更好，但没它也能交
   - **可选**：主要用于答辩增强
4. 等待用户确认清单

### 阶段二：批量生成
5. 用户确认后，一次性生成全部图表
6. 输出最终汇总包

### 核心原则
- 正文逻辑先于图表生成
- 图表必须服务正文，不允许脱离正文单独堆图
- 如果某张图不如文字清晰，保留文字，不强制生成

## 二、图位占位符规范

正文中的图位占位符格式：

```
图{章节}-{序号}【待生成】：{图表名称}
```

示例：
```
图4-1【待生成】：系统分层架构图
图4-2【待生成】：数据流转闭环图
图4-3【待生成】：三级风险分级对照表
```

生成图表后，将占位符替换为实际引用：
```
图4-1：系统分层架构图（来源：assets/charts/market/tech-architecture-v1.html）
```

## 五、可视化建议格式

章节末尾的「可视化建议」统一使用以下格式：

```markdown
## 可视化建议

| 位置 | 建议作图 | 类型 | 优先级 | 核心信息 |
|------|----------|------|--------|----------|
| 4.1.1 | 系统分层架构图 | HTML | 必做 | 四层架构（终端→业务→能力→数据）及关键技术栈 |
| 4.1.2 | 数据流转图 | HTML | 必做 | 居民→AI分析→工单生成→社工处理→统计反哺完整闭环 |
| 4.2.3 | 混合方案决策流程图 | HTML | 必做 | 规则引擎与大模型的协同决策路径及降级逻辑 |
| 4.3.2 | 三级风险分级对照表 | Python | 必做 | 高/中/低风险判定条件、处置方式、SLA要求对比 |
| 4.4 | 系统容错与安全架构图 | HTML | 建议做 | 容错设计三层保障 + 隐私合规全链路 |
```

### 字段说明

| 字段 | 说明 |
|------|------|
| 位置 | 对应章节的4.1.1、4.2.3等小节编号 |
| 建议作图 | 图表的中文名称，需足够具体 |
| 类型 | `HTML` 或 `Python`，决定生成工具 |
| 优先级 | `必做` / `建议做` / `可选` |
| 核心信息 | 一句话说明图表要传达的核心结论 |

## 六、生成优先级

当多个图表需要生成时，按以下优先级处理：

```
1. 结构型内容 → HTML优先
   - 架构图、流程图、关系图
   - 需要展示元素间关系的

2. 数据型内容 → Python优先
   - 市场规模、占比、对比
   - 需要精确数值展示的

3. 概念型内容 → MiniMax图片生成
   - 场景图、产品mockup
   - 暂未开通，标记待补充
```

## 七、输出检查清单

章节完成后，检查以下事项：

```
□ 章节文件已保存至项目根目录
□ 文件名符合 chapter-{序号}-{章节简称}.md 格式
□ 可视化建议已附在章节末尾
□ 每个可视化建议已生成对应文件
□ HTML图表已保存至 assets/charts/{分类}/
```
