# Rewrite Execution Protocol

> 本协议是 review-rewrite-protocol 的第三步"逐章修订"的具体执行指南。
>
> 核心目的：将"文体约束"（不能写成什么样）转化为"改写动作"（具体该怎么改）。
>
> 使用前提：已熟悉 output-standard.md 的八条基本规则和文体约束，已完成结构化 review。

---

## 一、改写前：识别病灶类型

改写前必须先读完全文，判断原文属于哪种"病"，再匹配对应改写方法。禁止不诊断就直接改。

### 六种典型病灶

| 病灶编号 | 病灶名称 | 症状特征 | 对应改写方法 |
|----------|----------|----------|--------------|
| T1 | 政策罗列型 | 堆砌政策文件、缺乏分析，只写"政策说了什么"不写"对本项目意味着什么" | 五层论证 |
| T2 | 提纲扩写型 | 大量"第一是...第二是...第三是..."，或bold标签式小标题，内容是提纲的展开而非论证 | 拆句扩写 + 去标签化 |
| T3 | 短句堆叠型 | 连续独立短句，缺乏论证展开，读起来像列表或PPT文案 | 合并重组 + 段落衔接 |
| T4 | 论证不足型 | 有论点但缺乏证据，结论跳跃，数据缺乏解释 | 先写厚再收敛 |
| T5 | 展示稿口吻型 | 含meta叙事、元说明、面向评委的提示，或"令人振奋"等宣传腔 | 去meta + 文风收敛 |
| T6 | 缺项目落点型 | 有一般性分析但无对本项目目标、节奏、指标的具体影响 | 补出项目层结论 |

### 病灶处理优先级

```
先处理T5（去meta/宣传腔）
→ 再处理T2（去标签/提纲结构）
→ 再处理T1/T4/T6（补论证/补落点）
→ 最后处理T3（段落衔接）
```

## 二、固定改写动作词表

改写时只能使用以下动词化动作，不得使用"优化"、"加强"、"改进"等模糊表述。

### 允许使用的改写动作

| 动作类别 | 适用场景 | 动作动词 |
|----------|----------|----------|
| 删除类 | 删除标签句、meta句、引导句 | 删除、移除、清去 |
| 合并类 | 合并短句、分点罗列 | 合并、整合、并入 |
| 改写类 | 改变表述方式 | 改写为、重构为、转化为 |
| 补出类 | 补出缺层论证 | 补出、增加、添加 |
| 重组类 | 调整内容顺序 | 前置、后接、重排 |
| 展开类 | 将判断句扩写为论证段 | 展开、延伸、铺陈 |
| 收束类 | 段末或篇末总结 | 收束、归结、点明 |
| 衔接类 | 建立段落间逻辑连接 | 增加承上句、增加过渡句 |

## 三、四栏制问题诊断模板

每发现一个问题点，必须按以下四栏逐项填写：

```
问题点编号：
原句：[摘录具体有问题的句子，不超过一句]
问题类型：[归入T1-T6之一，附上简述]
改写动作：[从第二章固定动作词表中选取，写明做什么]
协议依据：[引用本协议具体条款，说明为什么这样改]
```

- **问题类型**：T3 / 段落衔接不足 + T6 / 缺项目落点
- **改写动作**：在该措施前增加承上句"这一风险意味着我们必须主动缩短采购决策周期"；将"确保不受影响"改写为具有因果逻辑的表述

本段主问题：原文为典型的说明书式列项，用"第一级/第二级/第三级"和"场景一/场景二/场景三"结构分列技术方案，内容完整但呈展示型框架。
重写目标：删除枚举框架，合并为连续书面论述，保留原有技术信息密度。

**改写前原文**

系统设计了三级降级策略来应对各种异常情况。

第一级是性能降级，当大模型响应时间超过5秒时，系统自动切换到规则引擎模式，这个切换是透明的，居民和社工感知不到，工单会被标注"AI响应超时，数据仅供参考"。第二级是服务降级，当大模型API完全不可用时，系统切换到纯规则模式，此时系统功能受限，只能处理规则覆盖的场景，但核心的高风险识别功能仍然有效。第三级是兜底降级，当分类置信度低于60%时，系统增加追问轮次（最多4轮），如果追问后仍然置信度低，系统转人工处理，并将该案例标记为"待补充"供后续优化。

以下是几种典型异常场景的处理方式。场景一是网络异常，当检测到网络连接不稳定时，系统启动本地缓存机制，居民的描述被缓存到本地，网络恢复后自动同步到云端处理，社工端会显示"网络异常，处理可能有延迟"的提示。场景二是并发过高，当请求量突然增大、大模型可能出现排队时，系统自动启用请求分级，高风险请求优先处理，低风险请求短暂排队等待。场景三是模型版本更新，当底层大模型版本更新时，系统会先在测试环境验证，确认无误后再切换，切换过程对用户透明。

**改写后正文**

考虑到基层治理产品面向的是连续运行、风险敏感且对稳定性要求较高的真实服务场景，本项目在技术架构中设置了多级降级与容错机制，以确保系统在网络波动、模型异常、并发升高等情况下，仍能维持基本运行，并优先保障高风险识别与核心服务链路不被中断。其总体设计原则并非在任何情形下都维持完整智能能力，而是在异常条件下通过分层降级保住关键功能，从而避免因单点技术故障导致服务整体失效。

在具体机制上，系统采用三级降级策略。首先，当大模型响应时间超过预设阈值时，系统将自动切换至规则引擎处理模式，并对相关输出作出透明标记，提示社工该结果需结合实际情况进一步确认，以减少模型延迟对服务效率的影响。进一步地，当大模型 API 出现不可用情况时，系统将进入纯规则运行状态。此时，虽然复杂语义理解和精细分类能力会受到限制，但高风险识别和基础分类功能仍可保持有效，从而保证核心安全能力不因模型失效而中断。对于分类置信度过低、系统难以形成稳定判断的情形，平台则通过增加追问轮次补充信息；若在追问后仍无法获得足够可靠的判断依据，系统将自动转入人工处理路径，并将相关案例标记为"待补充"，用于后续回收分析与规则优化。

除模型层面的降级机制外，系统还针对若干典型异常场景设置了相应的容错方案。在网络异常情况下，平台通过本地缓存机制暂存居民输入内容，并在连接恢复后自动同步至云端，以降低瞬时中断造成的数据丢失风险；在并发请求短时升高的情况下，系统按照风险等级对请求进行优先级划分，优先保障高风险事项的分析、推送与响应；在模型版本更新时，则先在测试环境完成验证，再切换至正式环境，以减少底层升级对线上服务稳定性的冲击。通过上述设计，系统形成了面向真实治理场景的技术韧性，不仅能够提高平台连续运行能力，也能够为社工服务的稳定衔接和高风险事件的及时处置提供更可靠的技术支撑。

## 四、每段最低生成要求

改写后的每段必须同时满足以下六条：

1. **有明确论点句**：第一句亮出判断，不是现象描述
2. **有解释或证据**：不是只写判断，必须展开"为什么"
3. **有数据或案例**：论点必须有支撑，不能纯概念
4. **有落点**：段末必须有结论或过渡句
5. **无meta残留**：不能有任何面向写作者/读者的提示
6. **无标签残留**：不能有bold标签、numbered list等展示型结构

### 禁止的段落模式

```
❌ 判断句结尾（无论证）
"需求真实。"（没有展开）

❌ 列表罗列（展示型）
"三大核心优势：一、二、三。"

❌ 标语式短句（无论证）
"AI赋能，效率倍增。数据驱动，精准服务。"

❌ 跳跃式论证（缺环节）
"政策支持。技术成熟。市场很大。我们成功了。"（中间缺推导）

❌ 过度展开（论证冗余）
一个论点用500字举例三个同类案例

❌ 标签式引导句
"针对这一问题，我们采取以下措施。"
```

## 五、T1-T6 各病灶改写方法

### T1 政策罗列型 → 五层论证

**症状**：连续3段以上都是政策文件名称和内容摘录；没有"为什么这个政策对本项目重要"的论述。

**改写动作**：展开为五层
- 第一层（判断）：提出结论性观点
- 第二层（证据）：列明政策文件和数据
- 第三层（解释）：政策意味着什么（三个维度展开）
- 第四层（对本项目）：对项目切入时机、预算、决策周期的影响
- 第五层（收束）：归结为本项目的核心竞争优势

### T2 提纲扩写型 → 拆句扩写 + 去标签化

**症状**：大量"第一是...第二是..."或bold标签式结构。

**改写动作**：
1. 识别并删除标签式引导句（如"针对这一问题，我们采取以下措施"）
2. 将分点改为承接式段落
3. 用递进逻辑（不是并列逻辑）串联各项措施
4. 每项措施后补出"为什么有效"

### T3 短句堆叠型 → 合并重组 + 段落衔接

**症状**：连续独立短句，像顺口溜或PPT要点。

**改写动作**：
1. 识别相邻短句的共同主题
2. 合并为包含主语的完整复句
3. 用"——"或因果连词建立句内逻辑
4. 检查段与段之间是否有承上句

### T4 论证不足型 → 先写厚再收敛

**症状**：有论点但缺乏证据，结论跳跃，数据缺乏解释。

**改写动作**：
1. 针对每个判断追问"证据是什么"
2. 补出证据层
3. 再追问"这个证据对本项目意味着什么"
4. 最后收敛：删除冗余，保留核心论证

### T5 展示稿口吻型 → 去meta + 文风收敛

**症状**：含meta叙事、元说明、宣传腔。

**改写动作**：
1. 删除"评委看到..."、"评委会关注..."类meta叙事
2. 删除"本节用于说明..."类元说明
3. 将"令人振奋"、"重磅"等宣传腔改为中性陈述
4. 删除"早发现、早介入"等口号化表达
5. 用逻辑连接词替代"下面"、"首先"、"其次"等机械过渡

### T6 缺项目落点型 → 补出项目层结论

**症状**：有一般性分析但没有落到本项目目标、节奏、指标上。

**改写动作**：
1. 追问"这个判断对项目意味着什么"
2. 补写对年度目标、推进节奏、关键指标的具体影响
3. 用具体数字替代模糊表述

## 六、改写执行顺序（默认工作流）

改写任何章节时，按以下六步执行：

```
第一步：【诊断】通读全文，判断病灶类型（T1-T6）
↓
第二步：【排序】按T5→T2→T1/T4/T6→T3顺序处理
↓
第三步：【执行】按第三章匹配方法，对每个问题点执行"四栏制"改写
↓
第四步：【自检】每段满足"每段最低生成要求"六条
↓
第五步：【通读】全章改完后通读，检查段落衔接
↓
第六步：【输出】直接输出修订稿正文，不加任何meta说明
```
