# Rewrite Execution Protocol

> 本协议是 review-rewrite-protocol 的第三步"逐章修订"的具体执行指南。
>
> 核心目的：将"文体约束"（不能写成什么样）转化为"改写动作"（具体该怎么改）。
>
> 使用前提：已熟悉 output-standard.md 的八条基本规则和文体约束，已完成结构化 review。

---

## 一、改写前：识别病灶类型

改写前必须先读完全文，判断原文属于哪种"病"，再匹配对应改写方法。

禁止不诊断就直接改。

### 五种典型病灶

| 病灶编号 | 病灶名称 | 症状特征 | 对应改写方法 |
|----------|----------|----------|--------------|
| T1 | 政策罗列型 | 堆砌政策文件、缺乏分析，只写"政策说了什么"不写"对本项目意味着什么" | 五层论证 |
| T2 | 提纲扩写型 | 大量"第一是...第二是...第三是..."，或bold标签式小标题，内容是提纲的展开而非论证 | 拆句扩写 + 去标签化 |
| T3 | 短句堆叠型 | 连续独立短句，缺乏论证展开，读起来像列表或PPT文案 | 合并重组 + 段落衔接 |
| T4 | 论证不足型 | 有论点但缺乏证据，结论跳跃，数据缺乏解释 | 先写厚再收敛 |
| T5 | 展示稿口吻型 | 含meta叙事、元说明、面向评委的提示，或"令人振奋"等宣传腔 | 去meta + 文风收敛 |

### 常见混合类型

多数章节同时存在多种病灶。处理顺序：

```
先处理T5（去meta/宣传腔）→ 再处理T2（去标签/提纲结构）→ 再处理T1/T4（补论证）→ 最后处理T3（段落衔接）
```

## 二、T1 政策罗列型 → 五层论证

### 症状
- 连续3段以上都是政策文件名称和内容摘录
- 没有"为什么这个政策对本项目重要"的论述
- 没有数据和案例支撑

### 五层论证改写模板

原始内容示例：
```
国家近年来出台了一系列支持基层治理数字化的政策。2021年国务院发文要求加强基层治理现代化，2022年民政部推进智慧社区建设...
```

改写为五层：
```
第一层（判断）：政策红利窗口期已经到来，基层治理数字化具备顶层政策支撑。

第二层（证据）：2021年中共中央、国务院印发《关于加强基层治理体系和治理能力现代化建设的意见》，明确"到2035年基层治理现代化基本实现"；2022年民政部发布《"十四五"城乡社区服务体系建设规划》，要求提高基层治理智能化水平。这些政策构成自上而下的推力。

第三层（解释）：政策密集出台意味着三件事：一是基层治理数字化不再是可选项而是必选项；二是财政资金将持续向此领域倾斜；三是政府采购需求将在未来3-5年内集中释放。

第四层（对本项目）：这意味着本项目切入的时机恰好处于政策加速落地期，试点社区更容易申请到专项预算，采购决策周期也可能缩短。

第五层（收束）：综合以上，本项目应将"政策适配性"作为核心竞争优势之一，在申报书中重点着墨。
```

## 三、T2 提纲扩写型 → 拆句扩写 + 去标签化

### 症状
- 大量"第一是...第二是...第三是..."或"一是...二是..."
- Bold标签式结构如"**风险描述**"、"**应对措施**"
- 每段是提纲要点的展开，而非连贯论证

### 去标签化步骤

**Step 1：识别并删除标签式小标题**

原文中：
```
**风险描述**：政府采购周期长...
**影响分析**：导致Year2收入目标...
**应对措施**：采用分阶段签约...
```

改写为：
```
政府项目采购流程通常需要6-12个月，这一特点对需要快速起量的创业项目来说是不可忽视的时间成本。基层治理的采购主体主要是街道办和民政部门，这些单位的采购流程受到严格的财政预算和审计制度约束，从需求提出、立项审批、招标公告、评标定标到合同签订，整个周期往往需要半年甚至更长时间。如果政府采购周期过长，将直接影响Year2的收入目标，原本计划Year2签约的6个社区可能被迫推迟到Year3，导致Year2收入大幅缩水。

针对这一问题，我们采取四项措施：一是通过免费试点让社区提前体验产品价值形成使用习惯，等正式采购时决策周期大幅缩短；二是提前介入预算申报周期，在Year4季度就开始与目标社区沟通下一年度预算申报引导他们将产品纳入预算计划；三是采用"分阶段签约"模式，与社区签订框架协议明确年度服务内容和价格，到期自动续约减少每年重新招标的麻烦；四是建立多层级目标客户池，同时跟进10+个潜在社区确保即使部分社区采购延迟其他社区的进度不受影响。
```

**Step 2：拆句扩写**

原文中每一句判断，必须追问"为什么"，展开为完整论证：

原文（判断句，无展开）：
```
高风险识别误判或漏判可能带来不可挽回的后果。
```

拆句扩写后：
```
高风险识别是本产品的核心价值主张，漏判可能导致危机事件未能及时介入，误报则可能导致社工资源浪费或居民恐慌。对于本产品而言，一次严重的高风险漏判不仅伤害居民权益，也将终结项目前途——试点社区可能因此终止合作，团队信誉严重受损。同样，频繁的误报会让社工对系统失去信任，导致产品口碑崩塌。因此，对高风险管理必须格外审慎，不能仅依赖单一技术手段。
```

## 四、T3 短句堆叠型 → 合并重组 + 段落衔接

### 症状
- 连续3句以上独立短句
- 每句以逗号隔开，缺乏论证推进感
- 读起来像顺口溜或PPT要点罗列

### 合并重组方法

原（短句堆叠）：
```
多渠道融合推广是指在小区公告栏、楼道、电梯等醒目位置张贴推广海报，在物业通知、社区活动中嵌入产品介绍，与社区积极分子合作口碑传播。降低使用门槛是指开发语音输入功能方便不便打字的老年居民，优化交互流程确保3步之内完成诉求提交，提供"一键拨打"热线作为兜底。建立激励机制是指设计积分系统居民使用小程序可得积分积分可兑换小礼品或社区服务，定期公布处理进度让居民感受到被重视。
```

改（合并重组）：
```
针对居民使用率提升，我们设计了覆盖推广、体验、激励三个维度的综合方案。多渠道融合推广是指在小区公告栏、楼道、电梯等醒目位置张贴推广海报，在物业通知、社区活动中嵌入产品介绍，与社区积极分子合作口碑传播——通过物理触点和社区网络实现被动触达。降低使用门槛是指开发语音输入功能方便不便打字的老年居民，优化交互流程确保3步之内完成诉求提交，提供"一键拨打"热线作为兜底——让技术障碍较高的群体也能无门槛使用。建立激励机制是指设计积分系统居民使用小程序可得积分，积分可兑换小礼品或社区服务，定期公布处理进度让居民感受到被重视——通过价值感知提升持续参与意愿。
```

### 段落衔接检查

改完后通读，检查段与段之间是否有承上句。方法：每段第一句是否能接续上一段最后一句？

## 五、T4 论证不足型 → 先写厚再收敛

### 症状
- 有结论但论据单薄
- 数据列了但没有解释
- 逻辑跳跃，缺乏推导过程

### 操作步骤

```
Step 1：写厚（默认模式）
→ 针对每个判断，补出5层论证
→ 每个论点至少配2个证据
→ 允许冗余和重复

Step 2：收敛检查
→ 识别重复论述，合并同类项
→ 删除不服务核心观点的展开
→ 确保每个判断只论证一次

收敛标准：
□ 无重复论证（同一观点只论证一次）
□ 无冗余例子（每个证据只用一个）
□ 无过度展开（论证厚度与论点重要性匹配）
```

## 六、T5 展示稿口吻型 → 去meta + 文风收敛

### 症状
- 含meta叙事："评委看到的第X部分..."
- 含元说明："本节主要说明..."
- 含宣传腔："令人振奋"、"重磅"、"刚刚"
- 含面向读者的提示："评委会重点关注..."

### 去meta检查清单

改写时必须删除或改写：
- [ ] "评委看到..."、"评委会关注..."
- [ ] "本章/本节用于说明..."
- [ ] "下面"、"接下来"、"首先"、"其次"（机械过渡词）
- [ ] "令人振奋"、"重磅"、"刚刚"（宣传腔）
- [ ] "早发现、早介入、早化解"（口号化）
- [ ] "问题如下"、"建议"、"P0/P1/P2"（review词汇）

改写后，文风检查：
- 是否直接进入正文，没有引导语
- 是否用逻辑连接词而非机械过渡词
- 是否是陈述句而非感叹句

## 七、改写执行顺序（默认工作流）

改写任何章节时，按以下顺序执行：

```
第一步：通读全文，判断病灶类型（T1-T5）
↓
第二步：按优先级处理（先T5→T2→T1/T4→T3）
↓
第三步：每段改写后对照八条基本规则自检
↓
第四步：全章改完后通读，检查段落衔接
↓
第五步：输出前对照自检清单逐项检查
```

---

## 八、每段最低生成要求

改写后的每段必须同时满足：

1. **有明确论点句**：第一句亮出判断，不是现象描述
2. **有解释或证据**：不是只写判断，必须展开"为什么"
3. **有数据或案例**：论点必须有支撑，不能纯概念
4. **有落点**：段末必须有结论或过渡句
5. **无meta残留**：不能有任何面向写作者/读者的提示
6. **无标签残留**：不能有bold标签、numbered list等展示型结构

### 禁止的段落模式

```
❌ 判断句结尾（无论证）
"需求真实。"（没有展开）

❌ 列表罗列（展示型）
"三大核心优势：一、二、三。"

❌ 标语式短句（无论证）
"AI赋能，效率倍增。数据驱动，精准服务。"

❌ 跳跃式论证（缺环节）
"政策支持。技术成熟。市场很大。我们成功了。"（中间缺推导）

❌ 过度展开（论证冗余）
一个论点用500字举例三个同类案例
```

## 九、review-rewrite-protocol 补充条款

将以下内容追加到 review-rewrite-protocol.md 的"逐章修订"步骤中：

```
### Step 3. 逐章修订

每章必须先输出 review，再输出修订稿。修订稿执行步骤：

1. 【诊断】通读全文，判断病灶类型（T1-T5）
2. 【排序】按T5→T2→T1/T4→T3顺序处理
3. 【执行】按本协议第二章~第六章匹配方法改写
4. 【自检】每段满足第八章"每段最低生成要求"
5. 【通读】全章改完后通读，检查段落衔接
6. 【输出】直接输出修订稿正文，不加任何meta说明

改写必须调用 rewrite-execution-protocol.md 中的具体方法，
不得仅凭"文风约束"进行模糊修改。
```
