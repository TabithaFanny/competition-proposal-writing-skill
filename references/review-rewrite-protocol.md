# Review-Rewrite Protocol

## 适用场景
当用户要求：
- review
- 帮我检查
- 先review再改
- 逐章修订
- 系统处理全部章节
- 不要直接改，先做诊断

默认启用本流程。

## 执行顺序
### Step 1. 章节扫描
识别当前目录或当前任务范围内的全部章节，判断每章是否已有 review。

### Step 2. 补齐 review
对未完成 review 的章节，先补做结构化 review。

### Step 3. 逐章修订
每章必须先输出 review，再输出修订稿，之后才能进入下一章。

## review 的固定结构
1. 本章整体评价
2. 当前结构完整度
3. 各小节主要问题
4. 问题严重程度（高/中/低）
5. 缺数据、缺来源、缺逻辑、缺案例、缺衔接的具体位置
6. 修改优先级
7. 本章整体处理思路

## 修订原则
- 不是表面润色，而是结构性优化
- 扩写必须增加有效信息，不能注水
- 以逻辑链条、数据支撑、正式表达为优先
- 能重组结构则重组，能表格化则表格化
- 正式修订稿默认使用正式项目书文风
- 除非用户明确要求，否则正式稿必须段落化，不写成分点成稿
- 避免空泛套话、机械扩写、重复表述
- 数据必须尽量真实、具体、可核验
- 无法确认的数据必须标注"待核验"或"需补来源"

## 文体约束

正式修订稿默认采用项目申报书/正式计划书正文体，必须遵守以下规则：

1. review 可以结构化呈现，但修订稿必须是可直接进入正式文稿的连续正文
2. 除章标题、节标题外，默认不再继续添加提示性三级标题或标签式小标题
3. 禁止出现"章节定位""评委只会看几分钟""本节决定第一印象"这类面向写作者或评委的元说明
4. 禁止使用"场景切入""核心价值""三维痛点分析""竞品差异化优势"等演示型、提纲型小标题，除非用户明确要求保留这种结构
5. 禁止使用明显的提纲扩写句式，如"第一是……第二是……第三是……"，应改写为自然连贯的正式段落
6. 禁止写成短句堆叠、展示感过强、口号化明显的文本
7. 每一节正文应以完整段落展开，优先形成"论点—解释—证据—结论"的书面论证结构
8. 输出结果应让用户可以直接复制到项目书正文中，而无需再进行二次去提纲化处理

详细文体规范见：[output-standard.md](output-standard.md) 零-1节

## 输出格式

### review 输出（结构化）
```
# 第X章 Review报告

## 基本信息
- 章节：
- 字数：
- 完成度：

## 整体评价
[本章整体功能定位、核心问题]

## 问题汇总
| 位置 | 问题描述 | 严重程度 | 修改建议 |
|------|----------|----------|----------|
| 第X行 | ... | P0/P1/P2 | ... |

## 修订思路
[本章整体处理思路]
```

### 修订稿输出（段落化）
直接输出连续正文，不做任何说明，不输出 meta 叙事。

## review 粒度要求

每条 review 必须尽量具体，指向"如何改"：

```
❌ 错误示例（泛泛点评）：
"2.1.3 社会环境可以加强数据支撑"

✅ 正确示例（具体可执行）：
"第31行'21.1%老龄占比'，数据偏旧，应更新至2024年数据（约21.8%），
  来源：国家统计局2024年国民经济公报。如无法查到，改为'持续上升'
  或标注'待核实'，不能直接沿用2023年数据。"
```

## 两种输出模式的区分

| 模式 | 适用场景 | 要求 |
|------|----------|------|
| **分析稿 / 审阅稿** | review 报告、数据核验清单、图表清单 | 可以结构化、分点、列清单，方便审查 |
| **成稿 / 交付稿** | 正式比赛项目书正文 | 必须段落化、连续成文、自然过渡，少用 bullet points |
