# 章节深度写作指南 v2.1

> 每章必须覆盖的核心维度，避免只有骨架没有肉，或论证不充分。
>
> **v2.1新增**：章节扩充维度细化 · 先写厚再收敛执行标准

## 通用：字数密度要求（硬性）

### 最低字数标准

| 章节 | 最低字数 | 参考范围 |
|------|----------|----------|
| 第一章 执行摘要 | 3000字 | 3000-5000字 |
| 第二章 市场分析 | 8000字 | 8000-12000字 |
| 第三章 产品机制 | 8000字 | 8000-12000字 |
| 第四章 技术分析 | 8000字 | 8000-12000字 |
| 第五章 商业模式 | 5000字 | 5000-8000字 |
| 第六章 运行保障 | 5000字 | 5000-8000字 |
| 第七章 团队介绍 | 3000字 | 3000-5000字 |
| 第八章 财务分析 | 4000字 | 4000-6000字 |
| 第九章 风险分析 | 3000字 | 3000-5000字 |

### 为什么强调字数

比赛计划书不是摘要，不是提纲，每章必须有足够的论述密度。

- **评委视角**：计划书厚度体现项目认真程度，过于单薄会让评委觉得工作量不足
- **论证深度**：字数不够的地方往往是论证最薄弱的地方，扩写能强迫自己想清楚
- **竞争门槛**：进入决赛的项目各章节通常都在8000字以上

### 字数不足的常见错误

```
❌ 每节只有2-3段话就结束
❌ 大量使用列表（一级标题下列5个点）
❌ 结论和证据一样长
❌ 产品介绍章节比市场分析章节还短
```

### 什么叫有效扩写

扩写不是堆字数，是补论证。

**扩写前**（字数不足，论证跳跃）：
```
智微社工采用人机协同模式，提升社工效率。
```

**扩写后**（增加70%，论证完整）：
```
智微社工采用人机协同模式，而非纯AI替代社工。
这一设计选择基于以下判断：一线社工的核心价值在于人际沟通和复杂判断，
AI无法替代。问卷数据显示，71.33%社工认为"重复性问题"是最大痛点，
而非"AI取代自己"。因此，AI负责整理和识别，人负责判断和处置，
这种分工既能提升效率（AI减少60%的信息整理工作量），
又能保留人的核心价值（社工聚焦复杂问题处理）。
```

### 扩写质量检查

每小节写完后自问：
1. 这个判断的证据充分吗？有没有"因为X所以Y"中间缺一环？
2. 解释"意味着什么"了吗？评委不读脑子，需要写出来
3. 用了排除法吗？"选A不选B"的理由说清楚了吗
4. 有没有1句话超过3行的？如果有，拆开
5. 字数达标了吗？（按上表）

## 通用：每小节至少5层论证

### 标准论证结构（5层）

```
第1层：提出判断
→ "本项目选择聚焦社区社工AI辅助赛道"

第2层：给出证据
→ "根据300份问卷，90.33%社工对AI分类持正面态度"

第3层：解释证据意味着什么
→ "这说明需求不是想象出来的，而是来自一线真实痛点的积累"

第4层：说明对本项目的影响
→ "对于本项目而言，这意味着产品定位有真实的用户基础，而非闭门造车"

第5层：收束成结论
→ "因此，我们将目标市场锁定在这一群体，是需求验证后的理性选择"
```

### 错误示范（只有1-2层）
```
90%社工支持AI分类。（只有证据）
因此市场很大。（跳跃）

90%社工支持AI分类，这说明需求真实。（2层，但缺第4层）
```

### 正确示范（完整5层）
```
90.33%的受访社工对"AI辅助问题分类"持正面态度。（证据）
这一比例远高于我们项目启动前的预期，反映出一线社工对智能化工具的开放态度远超行业普遍认知。（解释）
这意味着产品的目标用户具备基本的技术接受度，用户教育的成本和阻力将显著低于预期。（对项目影响）
因此，本项目在产品推广阶段可以将"技术接受度"作为已验证优势，而非假设前提进行论证。（结论）
```

## 通用：至少2个横向维度

### 什么意思

每个小节的核心判断，不能只从一个角度论证，必须至少从2个不同维度交叉验证。

### 示例：目标市场选择

**单一维度（错误）**：
```
需求真实：90%支持 → 结论：选择这个市场
（只有一个证据，论证不充分）
```

**多维度（正确）**：
```
维度1-需求真实性：90%支持，71%认为重复性问题最大痛点
维度2-服务可交付性：产品机制与痛点高度匹配
维度3-付费可行性：街道办/民政局有预算采购
维度4-政策适配性：符合基层治理现代化方向
维度5-复制扩张性：长三角模式可推广
→ 结论：五个维度交叉验证，市场选择合理
```

## 通用：排除法思维

### 什么意思

成熟计划书不只是论证"为什么选A"，还要论证"为什么不选B/C"。

### 必须用排除法的场景

| 问题 | 正面论证 | 排除法 |
|------|----------|--------|
| 为什么选长三角 | 数字化基础好 | 为什么不选中西部的广大市场？（基础设施薄弱、采购周期长、团队资源有限） |
| 为什么选社工群体 | 痛点真实 | 为什么不选居民直接to C？（支付意愿低、使用频率低、需求分散） |
| 为什么用混合架构 | 规则保底+大模型增强 | 为什么不纯用大模型？（成本高、响应不稳定、缺乏可控性） |
| 为什么从B端收费 | B端付费能力强 | 为什么不从C端收费？（用户分散、付费意愿低、获客成本高） |

### 排除法写作模板

```
本项目选择[方案A]，而非[备选B/C]，主要基于以下判断：

第一，[排除B的理由]……因此暂不选择B。
第二，[排除C的理由]……因此暂不选择C。
第三，[选择A的独特优势]……这使得A是最适合当前阶段的方案。

这种排除法思考，确保我们的每个战略选择都经过了系统性的筛选，而非拍脑袋决定。
```

## 各章必须覆盖的维度清单

### 第四章 市场分析

必须覆盖8个维度：

| # | 维度 | 最低要求 | 充分展开 |
|---|------|----------|----------|
| 1 | 市场边界 | 定义清楚 | 说明边界外的为什么不选 |
| 2 | 用户分层 | 至少有2类用户 | 分层依据、每层特点 |
| 3 | 区域选择依据 | 说明选哪里 | 为什么不选其他地方 |
| 4 | 需求真实性 | 问卷/访谈数据 | 需求频率、严重度、可付费意愿 |
| 5 | 服务匹配性 | 产品能解决这个问题 | 机制与痛点的对应关系 |
| 6 | 支付路径 | 谁买单 | 采购链条、决策周期、付费意愿 |
| 7 | 竞争替代关系 | 与直接竞品的关系 | 互补还是替代、差异化在哪 |
| 8 | 扩张复制逻辑 | 试点到规模化的路径 | 复制成本、关键壁垒 |

**常见错误**：只写TAM/SAM/SOM数字，缺维度3、6、8

---

### 第三章 产品机制分析

必须覆盖8个维度：

| # | 维度 | 最低要求 | 充分展开 |
|---|------|----------|----------|
| 1 | 产品定位 | 一句话说明是什么 | 不是什么（排除法） |
| 2 | 核心产品 | 四个层面讲清楚 | 每层定位和价值 |
| 3 | 用户端设计 | 居民/社工/管理三端 | 每端的核心功能和价值 |
| 4 | 工作原理 | 完整流程讲清楚 | 从居民表达到问题解决 |
| 5 | 技术路线 | 规则引擎+大模型 | 混合方案优势 |
| 6 | 机制创新 | 至少3个创新点 | 前置识别/闭环服务/人机协同 |
| 7 | 性能指标 | 量化目标 | 每项指标的定义和依据 |
| 8 | 知识产权 | 布局情况 | 专利/软著/时间规划 |

**第三章常见错误**：

```
❌ 只写"产品功能清单"，不写"为什么这样设计"
❌ 技术路线和第二章重复，没有讲产品特色
❌ 创新点只有1个，或创新点和竞品优势混淆
❌ 性能指标只有数字，没有说明如何验证
❌ 居民端/社工端/管理端篇幅不均衡
```

**第三章扩写重点**（每个小节都要这样检查）：

- 3.1产品介绍：讲清楚"最终形态"是什么，四层面是什么关系
- 3.2工作原理：完整流程是什么，每个节点AI/社工/系统的分工
- 3.3机制优势：对三类用户（居民/社工/管理者）各有什么价值，要有数据支撑
- 3.4机制创新：前置识别/闭环/人机协同各是什么，为什么是创新
- 3.5性能指标：每项指标怎么测量，初始值目标值是多少

### 第五章 商业模式

必须覆盖6个维度：

| # | 维度 | 最低要求 | 充分展开 |
|---|------|----------|----------|
| 1 | 价值主张 | 一句话说明 | 与竞品的差异化定位 |

| 6 | 可规模化 | 能复制吗 | 规模化的关键驱动因素 |

必须覆盖5个维度：

## 什么叫有效扩写 vs 注水扩写

### 有效扩写（让论证更透）

```
原始：需求真实。
扩写后：需求真实——300份问卷显示，71.33%社工将重复性问题列为最大痛点，90.33%对AI分类持正面态度。这意味着产品的目标用户不仅存在，而且具备技术接受度基础。
```

**特征**：
- 每句话都有信息增量
- 解释"为什么"和"意味着什么"
- 不水字数

### 注水扩写（把句子拉长但没新信息）

```
原始：需求真实。
扩写后：经过深入的调研分析，我们发现一个令人振奋的事实——广大社区工作者对于智能化工具的引入，普遍持开放欢迎态度，这一发现极大地增强了我们对项目前景的信心。
```

**特征**：
- 大量情绪词（令人振奋、极大增强、广大）
- 没有实质新信息
- 像宣传稿不像项目书

## 禁用词清单（修辞过度）

### 宣传腔禁用词

| 禁用词 | 替代表达 |
|--------|----------|
| 令人振奋 | （直接陈述事实） |
| 极大增强 | 显著提升 |
| 广大社区 | 一线社工 |
| 碉堡了/太强了 | （禁止使用） |
| 颠覆性 | 具有创新性 |
| 革命性 | 实质性 |
| 绝对领先 | 差异化显著 |
| 无与伦比 | （禁止使用） |
| 完美解决方案 | 有效方案 |
| 极具商业价值 | 具有商业可行性 |
| 精准击中 | 针对性解决 |
| 绝佳试验田 | 合适的试点区域 |
| 坚实起点 | 良好的起步 |
| 一腔热血 | （禁止使用） |
| 闭门造车 | 脱离实际 |

### 学术腔但口语化禁用

| 禁用词 | 替代表达 |
|--------|----------|
| 那么 | （删除或改写） |
| 真的 | （删除） |
| 非常 | （用具体数据替代） |
| 特别 | （用具体描述替代） |

## 章节扩充维度细化（v2.1新增）

### 什么叫"展开充分"

一个章节展开充分的标准不是字数多，而是论证维度完整。

**自检问题**：
1. 这个章节回答了评委心里的核心问题吗？
2. 每个小节都有5层论证吗？
3. 有没有"维度缺失"？（如：市场分析缺区域选择依据）
4. 关键判断有没有段落化？
5. 有没有用排除法？

### 各章常见维度缺失

| 章节 | 高频缺失维度 | 补充方向 |
|------|------------|----------|
| 第二章 市场分析 | 区域选择依据、支付路径、复制扩张逻辑 | 为什么不选其他区域/如何从小区域复制 |
| 第三章 产品机制 | 机制创新解释、性能指标验证方式 | 为什么是创新而非改进 |
| 第四章 技术分析 | 技术选型对比、降级策略 | 为什么不选其他方案 |
| 第五章 商业模式 | 现金流健康度、盈利临界分析 | 回款周期多久、资金占用多大 |
| 第六章 运行保障 | 实施路径细节、激励机制 | 试点怎么选、如何复制 |
| 第七章 团队介绍 | 协作默契证明、梯队建设 | 之前合作过什么、扩展计划 |
| 第八章 财务分析 | 假设依据、敏感性分析 | 数据怎么来的 |
| 第九章 风险分析 | 应对策略可行性 | 具体怎么应对 |

### 扩充分辨率

```
完成度 < 60%：只写核心骨架 → 扩充到核心维度
完成度 60-80%：有骨架有血肉 → 补齐缺失维度
完成度 80-90%：基本完整 → 深化关键判断段落
完成度 > 90%：接近完成 → 收敛检查，删除冗余
```

### 先写厚再收敛的操作

```
阶段1：写厚（按5层论证+维度清单展开）
     ↓
阶段2：收敛检查
     ↓
     ├── 重复论述 → 合并/删除
     ├── 冗余例子 → 只保留最有力的
     └── 不服务核心观点的展开 → 删除

收敛后：每个判断只出现一次，每个证据只用一个
```

## 工作流v3：五阶段标准流程

### 阶段1：项目主档建立
**输出**：Project Master（唯一信息源，状态标注完整）

### 阶段2：Research补证据
**输出**：证据池（数据/政策/竞品/案例，不进正文）
**注意**：不补私有事实（团队经历、已有合同）

### 阶段3：分析稿模式
**输出**：结构化版本（便于Review）
- 章节结构
- 关键判断
- 数据点清单
- 缺失项
- 图表建议

### 阶段4：成稿模式
**输出**：正式比赛文书
- 连续正文
- 多维论证（每节≥5层）
- 排除法思维
- 克制文风
- **禁止宣传腔**

### 阶段5：交叉Review
**检查**：
- 数据是否支撑判断
- 用户画像是否一致
- 市场选择是否有排除法
- 商业逻辑是否闭环
- 文字是否克制

### 阶段6：主编统稿
**统一**：
- 术语
- 字数密度
- 段落节奏
- 风格
- 各章逻辑衔接
