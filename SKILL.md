# Competition Proposal Writing Skill - 比赛计划书写作助手

> **版本**: 2.3.0 | **更新时间**: 2026-03-24
> **核心架构**: 项目主档 → Research补证据 → 分析稿 → 成稿 → Review修订 → 统稿可视化
>
> **五大核心方向**: 持续执行但受控 · 先写厚再收敛 · 少表格多论证 · 章节交付闭环 · 图表升级答辩级

## 工作流v3.1（六阶段）

### 阶段0：建立项目主档
**输入**：用户项目基础信息
**输出**：Project Master（唯一信息源，状态标注完整）
**规范**：project-master.md

### 阶段1：Research补证据
**输出**：证据池（数据/政策/竞品/案例，不进正文）
**规范**：research-rules.md
**注意**：只补公开信息，不补私有事实
**触发**：主编判断是否需要WebSearch

### 阶段2：分析稿模式
**输出**：结构化版本（便于Review）
- 章节结构/关键判断/数据点/缺失项/可视化建议
**规范**：chapter-depth-guide.md（各章维度清单）

### 阶段3：成稿模式
**输出**：正式比赛文书
- 连续正文/多维论证/排除法/克制文风
**规范**：long-form-writing.md（5层论证/禁用词/过渡规范）

### 阶段4：Review与修订
**输出**：修订稿
- 主编协调交叉review，生成修订稿
**规范**：editor-in-chief.md

### 阶段5：统稿与可视化输出
**输出**：定稿 + 可视化图表
- 术语统一/口径统一/文风统一
- 根据章节需求决定可视化方案
**规范**：editor-in-chief.md + chart-style.md + html-prompt.md

## 两阶段输出流程（默认）

当用户说「第X章全部输出」时，按以下两阶段执行：

### 阶段一：正文先行
1. 读取项目配置文件 `.project-config.md`
2. 输出本章成稿正文，正文中用占位符标注图位：`图4-1【待生成】：xxx`
3. 同时输出可视化清单，标注每张图的优先级：
   - **必做**：没有它正文明显变弱
   - **建议做**：有了更好，但没它也能交
   - **可选**：主要用于答辩增强
4. 等待用户确认可视化清单

### 阶段二：批量生成
5. 用户确认后，一次性生成全部图表/HTML
6. 输出最终汇总包：
   - 正文终稿（含实际图号）
   - 图表文件清单
   - 图号与插入位置
   - 修订说明/缺失项说明

### 核心原则
- 正文逻辑先于图表生成
- 图表必须服务正文，不允许脱离正文单独堆图
- 如果某张图不如文字清晰，保留文字，不强制生成

## 核心规则

### 自主执行任务链协议（v2.3.0新增）

#### 状态机定义

```
P0待机 → P1执行中 → P2待确认 → P3完成 → P4阻塞
         ↑__________________________________↓
```

| 状态 | 含义 | 出口条件 |
|------|------|----------|
| P0待机 | 等待任务指令 | 收到明确任务 |
| P1执行中 | 正在执行任务 | 任务完成/遇硬性缺失/用户叫停 |
| P2待确认 | 需要用户决策 | 用户确认/用户否决/用户修正 |
| P3完成 | 任务交付完成 | 进入下一任务 |
| P4阻塞 | 硬性缺失无法推进 | 用户补充/跳过 |

#### 自主执行条件（同时满足才自动执行）

1. **任务明确**：用户给出了具体的章节号或任务描述
2. **信息充分**：主档已建立且核心字段完整
3. **无硬性缺失**：不触发"硬性缺失处理"规则
4. **上下文清晰**：在主档→成稿→Review→修订的正常流程内

#### 防跑偏规则

```
触发"叫停确认"的条件（任一）：
□ 任务目标与主档核心定位产生矛盾
□ 发现关键数据与已有结论冲突
□ 用户明确说"停"/"不对"/"不是这个方向"
□ 单次输出超过5000字未进行收敛
□ 连续3次迭代仍未达到质量基线

叫停后的处理：
→ 输出"当前状态报告"（已完成/进行中/问题点）
→ 列出2-3个可能的方向选项
→ 等待用户明确下一步指令
```

#### 持续执行模式（默认）

当用户说"继续"或"进行扩充"时：
1. 识别当前章节的完成度
2. 按照chapter-depth-guide展开未达标的维度
3. 先写厚（按5层论证展开），再收敛（删减冗余）
4. 不需要每次都问"要不要继续"

#### 任务链传递规则

```
当一个任务自然完成后：
→ 自动进入下一个逻辑任务（不等待）
→ 输出"上一任务完成状态 + 下一任务开始"
→ 例如："第三章第一稿完成，开始第二章扩充"

当任务链中出现断点：
→ 识别缺失内容
→ 列出需要补充的信息
→ 等待用户补充后继续
```

### 信息源唯一原则
```
所有章节内容必须从项目主档提取
research结果作为数据支撑
禁止编造主档中没有的信息
引用格式：📌 主档-[字段名]
```

### 硬性缺失处理
```
以下任一缺失时，主编停止分配任务，用户补充：
□ 市场规模（TAM/SAM/SOM）
□ 至少一个竞品名称
□ 核心团队成员（创始人+联创）
□ 商业模式概述
```

### research可自动/禁止清单
```
可自动search：
□ 市场规模和行业数据
□ 政策文件和官方报告
□ 竞品公开信息
□ 公开案例和获奖项目
□ 免费可商用图片

禁止自动research：
□ 团队成员真实履历
□ 已有合作协议
□ 已有用户数据
□ 融资进展
□ 核心技术细节
```

### WebSearch API配置
```
当前可用方案：Bocha API
- API地址：https://api.bochaai.com/v1/web-search
- 环境变量：BOCHA_API_KEY
- 使用说明：运行时从环境变量读取，不写入任何skill文档或仓库文件

注意：MiniMax API的WebSearch端点（/v1/web_search）返回400错误，
如需切换其他搜索服务，需同时更新API配置。
```

### 可视化三路分流规则（默认自动分流）

**默认规则：**
1. **数据型内容** → 默认走Python
2. **结构型内容** → 默认走HTML
3. **概念型/场景型内容** → 默认走图片生成
4. **文字已足够清晰时** → 不强制做图
5. **两种方案都合理时** → 哪个高端选哪个

### 可视化流程（新版：线框→高保真）

```
第一步：判断图的目标
    这张图是讲结论/关系/差异/路径？

第二步：输出低保真结构方案
    页面标题 + 视觉中心 + 信息分区 + 元素数量 + 版式草图说明 + 设计理由

第三步：用户确认后，输出高保真版本
    HTML单页 / Python图表 / 图片
```

**规范文件：**
- 分流规则：chart-style.md
- 质量标准：visualquality-guide.md
- 设计规范：presentation-design-guide.md

#### 分流决策树

```
发现可视化需求
     │
     ▼
┌─────────────────────────────────────┐
│ 核心信息是"精确数据"吗？              │
│ （数值比较、趋势、占比、预测）          │
└─────────────────────────────────────┘
     │是                    │否
     ▼                      ▼
Python matplotlib      ┌─────────────────────────────────────┐
                        │ 核心信息是"结构关系/流程逻辑"吗？     │
                        │ （架构图、流程图、闭环图、路线图）     │
                        └─────────────────────────────────────┘
                              │是                │否
                              ▼                  ▼
                        HTML单页            MiniMax图片生成
                        (assets/charts/)    (assets/images/)
```

#### 第一路：Python matplotlib（数据图表）
- **适用于**：市场规模、TAM/SAM/SOM、增长率、占比、趋势、预测
- **输出**：`assets/charts/*.png`
- **规范**：`chart-style.md`

#### 第二路：HTML单页（结构图/流程图）
- **适用于**：商业模式闭环、服务流程、产品架构、战略里程碑、组织结构
- **输出**：`assets/charts/*.html`
- **规范**：`html-prompt.md`
- **流程**：先提方案→用户确认→生成HTML

#### 第三路：MiniMax图片生成（概念图/场景图）
- **适用于**：概念示意、产品mockup、场景图
- **输出**：`assets/images/*.png`
- **状态**：可选通道，API配置后启用

#### 章节写作中的可视化建议格式

每章写作后固定附带：

```
## 可视化建议

| 位置 | 建议作图 | 类型 | 核心信息 |
|------|----------|------|----------|
| 4.2.1 | TAM/SAM/SOM规模图 | Python | 市场规模层级 |
| 4.1.3 | 市场定位三角图 | HTML | 定位要素关系 |
| - | 无需配图 | - | 文字已清晰表达 |
```

#### 竞品对比可视化方案验证

| 对比类型 | 推荐方案 | 理由 |
|----------|----------|------|
| 竞品功能矩阵（多维度评分对比） | Python | 精确数值对比，矩阵/柱状更直观 |
| 竞品定位象限图 | Python | 2x2象限，数值轴精确 |
| 竞品关系结构图 | HTML | 展示竞品间关系、定位差异 |
| 竞品优劣势对比 | Python | 表格/柱状更清晰 |

**结论：** 竞品对比优先Python（矩阵/柱状），仅当需要展示"竞品定位关系"时用HTML。

### 文风统一原则（详见tone-guide.md）
```
□ 专业、简洁、克制
□ 数据驱动，不空喊口号
□ 统一术语：项目名称/目标用户/核心功能
□ 禁止：颠覆/革命性/绝对领先/最优秀
□ 角色影响分析视角，不影响最终成文风格
```
